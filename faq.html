FAQ Installation & updates

DiscVault is designed to run self-hosted. The easiest setup is the all-in-one Docker container, which includes the web app and backend in one image. On Unraid, use the Community Applications template and keep your database and uploaded assets on persistent storage.

To install DiscVault 26, use the v26 image tag. To stay on the old Legacy version, use the legacy tag instead of latest.

GitHub Container Registry Unraid template

Before you update

  • Create a backup from the admin tools or copy the persistent data directory while the container is stopped.
  • Check the release notes when moving between beta and stable builds.
  • Keep your environment variables and volume mappings unchanged unless the release notes say otherwise.

Updating

Pull the newer image, recreate the container with the same volumes and settings, then open DiscVault and confirm that your collection, users, passkeys, backups, and settings are still visible. If anything looks wrong, stop the container and restore the backup before continuing.

FAQ Staying on the old version

If you are not ready to move to DiscVault 26 yet, you can stay on the previous DiscVault generation by using the legacy Docker image tag.

Use ghcr.io/helmerzNL/DiscVault:legacy instead of latest or beta in your Docker, Docker Compose, or Unraid template configuration. Keep your existing volume mappings, environment variables, ports, and persistent data path unchanged.

Recommended steps

  • Create a backup before changing the image tag.
  • Change only the image tag to legacy.
  • Pull the image, recreate the container, and confirm that your collection opens normally.

The legacy image is meant for people who want to keep running the old version. New DiscVault 26 features, migration improvements, and future interface updates are added to the v26 image instead.

FAQ Passkeys versus passwords

DiscVault uses passkeys so you do not need to create, remember, reuse, or share a password. A passkey is a cryptographic key pair: the private key stays on your device or in your trusted credential manager, while DiscVault only receives the public key.

That makes passkeys safer than passwords. There is no password database for an attacker to steal, nothing useful to type into a fake login page, and every passkey is unique to the site where it was created. Signing in is also easier because you approve it with your device unlock method, such as Face ID, Touch ID, fingerprint, Windows Hello, or a device PIN.

Why they are safer and easier

  • Phishing resistant: the browser and operating system only use the passkey for the correct website.
  • No shared secret: DiscVault stores a public key, not a reusable password.
  • Unique per service: a passkey created for DiscVault cannot be reused somewhere else.
  • Fast sign-in: unlock your device instead of typing a password.

Minimum requirements

  • Windows: use a current browser with passkey support and Windows Hello. Windows 11 version 22H2 or newer is recommended for native passkey management; Windows Hello needs at least a PIN, with face or fingerprint optional.
  • macOS: macOS Ventura 13 or newer with iCloud Keychain enabled, or another passkey-capable credential manager/browser.
  • iPhone and iPad: iOS 16 or iPadOS 16 or newer with iCloud Keychain and two-factor authentication enabled.
  • Android: Android 9 or newer with a passkey-capable credential provider such as Google Password Manager, plus a screen lock.

Tip: register more than one passkey where possible, for example your phone and your desktop device. That gives you a recovery path if one device is replaced or lost.

FAQ Why passkeys need a domain name (FQDN)

Passkeys use the WebAuthn standard, which binds each passkey to the exact web address where it was created. That address must be a real domain name — a fully qualified domain name (FQDN) such as discvault.example.com — served over HTTPS. A bare IP address like http://192.168.1.50:6080 does not work for passkeys.

When you open DiscVault, the browser ties every passkey to the relying party ID (RP_ID), which is the hostname shown in the address bar. The browser only offers a passkey when that hostname matches, and it refuses to create or use passkeys on a bare IP address or over plain HTTP. If you reach DiscVault by IP, the passkey step fails or never appears.

Why a domain name is required

  • Phishing protection: WebAuthn locks each passkey to one domain name, and an IP address cannot be validated as a domain, so browsers refuse to use it.
  • Secure context: browsers only allow passkeys over HTTPS or on localhost, and a trusted HTTPS certificate is issued for a domain name, not for an internal IP.
  • Stable identity: if your server IP changes, passkeys bound to that IP would break, while a domain name keeps working.

How to set it up

  • Point a domain or subdomain such as discvault.yourdomain.com at your server and put it behind HTTPS, usually through a reverse proxy.
  • Set RP_ID to that hostname without scheme or port, and set RP_ORIGINS to the full HTTPS URL, for example https://discvault.yourdomain.com.
  • Open DiscVault through the domain name from now on, then create your passkey. The localhost address still works on the server itself for first-time setup.

Tip: choose the domain name before creating passkeys. Passkeys are tied to the hostname, so changing it later means registering your passkeys again.
